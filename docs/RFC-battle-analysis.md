## TL;DR(先講結論)

比賽只看三件事:**(1) 明天就能用、(2) 價值能量化、(3) 比賽後還活著**;而且明文說「問題特不特別」「技術炫不炫」**都不加分**。

用這把尺量這份 RFC,會得到一個尖銳的落差:

- **RFC 的強項(架構優雅、CMP/Hot Reload、解一個真痛點)恰好踩在「不加分」的格子裡**,或需要很長的前置投入才兌現價值。
- **RFC 自己承認兩個 deal-breaker 尚未解(E9 AI 可靠性、P4+D1 designer 願不願轉職能),且 ROI 未量化(P1)** —— 這三點正好直接打在比賽的三個評分軸上。

**核心建議:不要把整份 RFC 當參賽作品。** 把它拆成兩半,只拿「明天就能用」的那一半(AI 規範檔 + hooks + component catalog/design token)去比賽,把需要大遷移的那一半(CMP common module + desktop target)留作後續 roadmap。詳見 [§5 如何提高得分](#5-如何提高得分把提案調成比賽會喜歡的形狀)。

## 1. 評分標準回顧(這場比賽真正在量什麼)

| # | 標準 | 白話 | 對本提案的關鍵問句 |
| --- | --- | --- | --- |
| 1 | **能不能實用** | 團隊明天就打開來用嗎?嵌進真實日常流程? | designer/RD 明天就能在現有 repo 用嗎,還是要先做完 module 遷移? |
| 2 | **有多大價值** | 省多少時間/成本?影響多少人/案件?要能**量化** | 一個月省下幾張 UI ticket 的 handoff?影響幾個 designer/RD? |
| 3 | **能不能常駐迭代** | 比賽後還活著嗎?能持續被用與改進? | 不靠某個人推就會自然續用嗎?designer 真的會天天開嗎? |

> ⚠️ 明文**不加分**:問題普不普通、技術炫不炫。
> 換句話說:**樸實但天天有人用、價值可量化、能長期活下去 > 炫技 demo。**

這把尺對本提案非常不友善的地方在於:RFC 花最多篇幅論證的「為什麼這樣選(CMP vs Jetpack Compose、Hot Reload 原理、KMP 依賴鐵律)」**在比賽裡一分都拿不到**。

## 3. 逐項評分對照

### 標準 1:能不能實用(明天就能用?)

**現狀評估:⚠️ 偏弱 —— 這是本提案最大的風險。**

- RFC 通篇是 **DRAFT / FOR DISCUSSION**,大量 Open Questions 未解,且自承有兩個 **deal-breaker 未答(E9、P4+D1)**。比賽要的是「明天打開就用」,提案卻停在「該不該啟動 pilot」的更前面一步。
- 「明天能用」的最大障礙是**前置遷移成本**:UI 層解耦、module 切分、導入 CMP + desktop target、DI 改造(E7)、CI/Gradle 成本(E8)。在這些鋪好之前,designer 根本開不了那個「會動的 Figma live editor」。
- RFC 自己也說「建議先以單一 feature 做 pilot,用數據驗證後再擴散」—— 這是負責任的工程態度,但對一個**以「明天就用」為第一評分軸**的比賽,等於承認「現在還不能用」。

**但有一條捷徑**:提案裡有一塊**不需要任何架構遷移、今天就能裝進現有 repo**的東西 —— **AI 規範檔(Claude Skill / `CLAUDE.md` / rules)+ Claude Code hooks + component catalog/design token 約束(L1+L2)**。這塊本身就能讓「AI 生 UI code 守 design system」在現有 codebase 立即生效,不必等 CMP。**這才是比賽該主打的「明天就能用」核心。**

### 標準 2:有多大價值(能量化?)

**現狀評估:○ 中等 —— 方向對,但目前是「待驗證的假設」而非「已兌現的數字」。**

- 加分點:RFC 有完整的 **Success Metrics 段**(吻合度評分、UI 返工次數、一次生成可用率、迭代時間、APK/build diff…),代表作者**知道要量化**,符合評分軸 2 的精神。
- 扣分點:這些全是 **pilot 之後** 才有的數字。**P1 明白寫「ROI 與時程未量化、短期甚至可能拖慢交付」**。比賽要的是「省了多少、影響多少人」的**現在式數字**,提案給的是未來式承諾。
- 價值兌現的時間軸太長:大遷移 → pilot → 量測 → 擴散。比賽週期內很難跑出有說服力的價值數字。

**提高價值說服力的關鍵**:把「影響面」算清楚並前置 —— 例如「BU 內 X 位 designer、每月 Y 張 UI ticket、每張 handoff 來回平均 Z 小時」,先把**可省的總量天花板**估出來,再用一個小 pilot 量「實際省了多少」。

### 標準 3:能不能常駐迭代(比賽後還活著?)

**現狀評估:◎ 設計上強,但 ⚠️ 依賴一個高風險假設。**

- 結構性加分:**產出即 production code、進 repo、走 CI、靠護欄維持** —— 這是天生的「常駐」設計。RFC 反覆強調的 **anti-drift**(東西只有一份)本質上就是「不會腐爛、會一直被用」的論述,**完美對上評分軸 3**。這是提案對這場比賽最契合的一點,應重點放大。
- 致命依賴:常駐的前提是 **designer 真的願意改職能、天天用(P4 + D1 deal-breaker)**。只要 designer 不買單,整套就停用 → 比賽後立刻死亡。**P4 寫得很直白:「若 designer 不買單,所有工程投入打水漂。」**
- 次要依賴:E9(AI 能否穩定產出守規範的 Compose)。引擎不穩,designer 用兩次就放棄。

## 4. 評分總覽

| 評分軸 | 評估 | 一句話 |
| --- | --- | --- |
| 1. 實用(明天用) | ⚠️ 弱 | 全量版要先做大遷移,且 deal-breaker 未解;**但 L1+L2 那塊可立即用**。 |
| 2. 價值(量化) | ○ 中 | 有 metrics 意識,但數字是未來式,ROI 自承未量化。 |
| 3. 常駐(會活著) | ◎/⚠️ | 架構天生常駐(anti-drift 很契合),但卡在 designer 採用意願這個 deal-breaker。 |
| (不加分) 技術炫度 | — | CMP/Hot Reload/KMP 論證精彩,**但比賽不給分** —— 投入產出錯配。 |
| (不加分) 問題特別度 | — | 真痛點,但「特不特別」不影響分數。 |

**一句話總評**:這是一份**優秀的工程 RFC**,但**不是一份為這場比賽量身的參賽提案**。它的重心(架構正確性、技術選型)落在比賽不計分的區域,而比賽最看重的「明天就用 / 量化價值」正好是它最弱、最未完成的部分。

## 5. 如何提高得分(把提案調成比賽會喜歡的形狀)

### 建議一(最重要):拆成兩半,只拿「能立刻用」的那半參賽

RFC 其實含兩個可分離的部分:

| | 內容 | 對比賽的契合度 |
| --- | --- | --- |
| **A. 生成端約束(輕)** | Claude Skill / rules + Claude Code hooks + component catalog / design token 檢查 + 自訂 lint(Detekt) | **三軸全中**:今天就能裝進現有 repo(實用)、可立即量「AI 合規率/省下的 review 來回」(價值)、commit 進 repo 全 team 共用(常駐) |
| **B. 架構端(重)** | CMP common module + desktop target + Hot Reload + module 切分 | 大遷移、長前置、deal-breaker 未解 —— 對比賽三軸都偏弱 |

**把 A 當參賽作品,把 B 寫成「已驗證後的下一步 roadmap」。** A 不需要動架構、不需要 designer 先轉職能,RD 明天就能在現有 codebase 啟用 hooks 與規範檔,馬上產生「天天在用」的事實。

### 建議二:把價值換成現在式數字

- 先估**總量天花板**:BU 內 designer/RD 人數 × 每月 UI 相關 ticket 數 × 每張 handoff/來回釐清平均耗時。
- 再用 1–2 週小範圍實測**實際省下的時間**與 **AI 一次生成可用率**,做成 before/after 對照。
- 強調「影響多少人、多少案件」,這是評分軸 2 的原話。

### 建議三:重寫對外 pitch,砍掉不加分的論述

- **不要**用 CMP vs Jetpack Compose、Hot Reload 原理、KMP 依賴鐵律當賣點 —— 這些一分不拿,還稀釋評審注意力。
- pitch 結構直接對齊三軸:**「誰明天會用它(實用)→ 一個月省多少(價值)→ 為什麼它不會死(常駐)」**。
- 「產出即 production code、進 repo、靠 CI 護欄存活」這條 anti-drift 故事**正中常駐軸**,要放在最前面講。

### 建議四:在比賽前先處理(或誠實標記)deal-breaker

- **E9(AI 可靠性)**:用建議一的 A 部分,當場展示「有 skill/hooks vs 沒有」的合規率差異 —— 把 deal-breaker 變成 demo 亮點。
- **P4+D1(designer 採用)**:若比賽週期內無法讓 designer 轉職能,就**把參賽範圍限縮在 RD 端的生成約束**(RD 用 AI + 護欄產 UI code),避開採用風險,先把流程跑起來、產出數字,designer 轉型留作後續。

### 建議五:用「樸實天天用」框架包裝

比賽明說「樸實但天天有人用 > 炫技 demo」。把作品定位成**「一套讓 AI 產 UI code 不走鐘的 repo 內護欄,RD 每次寫 UI 都會經過它」**,比定位成「用 CMP 重構 UI 層的宏大工程」更討喜、也更可信。

---

## 6. 風險提醒(若堅持拿全量版參賽)

- **「還沒能用」會直接壓低第 1 軸**:評審若問「現在能開來用嗎」,答案是「要先做完遷移 + pilot」,這在三軸裡最致命。
- **價值是未來式**:沒有現在的數字,第 2 軸只能拿到「有規劃」的分,拿不到「已兌現」的分。
- **押注在採用意願**:第 3 軸的常駐優勢被 P4 抵銷 —— 架構再會活,人不用就是死。
- **投入產出錯配**:最花力氣的技術論證落在不計分區,等於把彈藥打在沒有靶的方向。

---

## 7. 結論

這份 RFC 作為**工程提案**是成熟、誠實、論證完整的(連自己的 deal-breaker 都標出來)。但作為 **EDU Flow Battle 的參賽作品**,它與評分尺的契合度是「中下」,主要因為:重心放在不計分的技術正確性,而最計分的「明天就用 / 量化價值」恰是它最未完成的兩塊。

**最高槓桿的調整是建議一**:把輕量的「AI 規範檔 + hooks + component catalog 護欄」抽出來單獨參賽 —— 它今天就能用(軸1)、能立刻量合規率與省下的 review(軸2)、commit 進 repo 全 team 共用且自然續用(軸3),完整命中三軸,又完全避開 CMP 大遷移與 designer 採用這兩個拖累。把宏大的 CMP/Hot Reload 願景留作「驗證成功後的下一步」,反而能讓參賽版本更樸實、更可信、更容易在比賽後活下去。
