Android Build Workflow

ClassSwift Teacher App  ·  APK Generation & QA Distribution

Staging Release Candidate Production
🎯 Purpose

This workflow is scoped to APK generation only. It builds signed APKs for the stag rc prod environments and distributes them to the QA team via S3 and email.

⚠️
Current limitation: All environments share the same staging S3 bucket (classswift-android-apk-aps1-stg). Dedicated RC and production buckets are not yet wired up. There is no CD step — APKs are delivered to QA for manual installation.
⚡ Trigger

The workflow is triggered manually via workflow_dispatch with the following inputs:

InputTypeRequiredOptions / DefaultDescription
environment choice Yes stag · rc · prod · all Target environment. all triggers a parallel matrix build for all three.
is_auto_upgrade_version boolean No Default: false When true, auto-bumps the version before building.
stag  increments internal versionCode by 1.
rc  increments minor version on develop.
is_test_mode boolean No Default: false When true: checks out the triggering branch instead of the env branch; skips git push and cherry-pick to develop; uploads APKs to the Test/ S3 prefix; prefixes the email subject with [CI Test].

📋 Job Details

Job 1 — prepare

Runs on ubuntu-latest. Resolves the build matrix from trigger inputs and outputs a JSON array (e.g. ["stag","rc","prod"]) consumed by the build job.

Job 2 — build (matrix)

Runs on ubuntu-latest, timeout 30 min. Executed once per environment resolved by prepare.

#StepConditionNotes
1Determine build config—Sets Gradle task, S3 prefix, branch, secret suffix, AWS role/region. Computes effective_prefix — Test when is_test_mode=true, otherwise the env prefix.
2Checkout—Full history (fetch-depth: 0). Checks out the triggering branch when is_test_mode=true, otherwise the env-specific branch.
3Check develop for version bumprc auto_upgradeReads latest commit on develop; sets skip_version_bump=true if bump already exists.
4Bump internal versionCodestag auto_upgradeIdempotency check, then increments internal counter (+1), commits locally. Not pushed in test mode.
5Compute version name—Derives MAJOR.MINOR.HOTFIX.INTERNAL from version.properties.
6Setup Java 17—Uses actions/setup-java@v4 with Temurin distribution.
7Setup Android SDK—Uses android-actions/setup-android@v3.
8Cache Gradle packages—Caches ~/.gradle/caches and ~/.gradle/wrapper keyed by Gradle file hashes.
9Create keystore files—Decodes JKS_BASE64 and KEYSTORE_PROPERTY_BASE64 secrets into files.
10Create google-services.json—Places per-flavor JSON under app/src/{device}{Env}/ for both aosp and edla.
11Build APKs—Runs the env-specific Gradle task; output to ~/Apks.
12Upload APKs as artifact—Stored 30 days under {env}-apks-{run_number}.
13Configure AWS credentials—Assumes IAM role via OIDC (id-token: write permission).
14Upload APKs to S3—Recursive upload to s3://classswift-android-apk-aps1-stg/{effective_prefix}/. Uses Test/ prefix in test mode.
15Push bump + cherry-pick to developstag auto_upgrade + has_bump + not testPushes bump commit to staging, cherry-picks onto develop. Skipped when is_test_mode=true.
16Bump minor version on developrc auto_upgrade + not skip + not testIncrements MINOR by 1, resets HOTFIX and INTERNAL to 0, pushes to develop. Skipped when is_test_mode=true.
17Print APK download URLs—Lists URLs via CloudFront at android-ota.aps1.classswift-stg.com/{effective_prefix}.
18Extract JIRA ticket IDsstagFinds the two most recent ci: update version commits on the target branch. Extracts VSFT-XXXX IDs from conventional commit scopes in that range. Outputs a deduplicated list.
19Fetch JIRA ticket titlesstag ticket_ids foundFor each ticket ID calls GET /rest/api/3/issue/{id}?fields=summary with Basic Auth. Falls back to ticket ID as title if the API call fails.
20Build email body—Generates an HTML email with per-APK download links. Injects a JIRA ticket table when step 19 produced results.
21Send email—Uses dawidd6/action-send-mail@v3 with SMTP credentials. Prefixes subject with [CI Test] when is_test_mode=true.
🏷 Version Scheme

Version is encoded as a single integer in version.properties:

FieldFormulaExample (versionCode = 1013000001)
MAJORversionCode / 100000001
MINOR(versionCode / 100000) % 10013
HOTFIX(versionCode / 1000) % 1000
INTERNALversionCode % 10001

Resulting version name: 1.13.0.1

🔑 Repository Secrets

This workflow reads sensitive values from GitHub repository secrets — encrypted variables stored at the repository level and injected into the workflow at runtime via secrets.SECRET_NAME.

⚠️
Secret values are never echoed in workflow logs. Anyone with write access to the repository can use secrets in workflows, but only repository admins can view, update, or delete them.

How to add a repository secret

  1. Open the repository on GitHub and click the Settings tab.
  2. In the left sidebar, expand Secrets and variables and click Actions.
  3. Under Repository secrets, click New repository secret.
  4. Enter the exact secret Name (case-sensitive, must match the table below) and paste the secret Value.
  5. Click Add secret. The secret is immediately available to all workflow runs.

Secrets used by this workflow

GOOGLE_SERVICES_JSON_EDLA_PROD prod Firebase config for the EDLA device flavor.

Generating secret values

Several secrets require base64-encoding a local file before pasting its value into GitHub. The commands below encode the file and copy the result directly to your clipboard via pbcopy (macOS).

Keystore

# JKS_BASE64 base64 -i MVBA_PlatForm.jks | pbcopy # KEYSTORE_PROPERTY_BASE64 base64 -i keystore.properties | pbcopy

Firebase google-services.json — run once per flavor + environment. Use the google-services.json downloaded from the matching Firebase project.

# GOOGLE_SERVICES_JSON_AOSP_STAG (repeat the same command for each of the 6 variants) base64 -i google-services.json | pbcopy

EMAIL_USERNAME and EMAIL_PASSWORD are plain-text strings — paste them directly, no encoding required.
